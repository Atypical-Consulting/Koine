# C# Emitter — Kind-Based Output Folders

**Date:** 2026-06-16
**Status:** Approved (design)
**Scope:** `src/Koine.Compiler/Emit/CSharp` — output file layout only

## Problem

The C# backend emits one file per type and groups them by namespace, where the
folder mirrors the namespace (`FolderFor(ns) => ns.Replace('.', '/')`). Within a
bounded context every file lands flat at the context root:

```
Catalog/Product.cs
Catalog/Price.cs
Catalog/Availability.cs
Catalog/IProductRepository.cs
...
```

For a context with dozens of types this flat list is hard to scan. We want the
generated tree to communicate the DDD building blocks at a glance.

Note: flat-folder-per-context is itself the .NET-idiomatic layout (folder ==
namespace). This change deliberately trades that convention for browsability of
*generated* code. It is safe to do so because every emitted file carries the
`// <auto-generated/>` header, which exempts it from analyzers such as IDE0130
(namespace-must-match-folder).

## Decision

Introduce a **kind subfolder** between the namespace folder and the file name.
The **namespace is unchanged** — it remains the bounded context (plus module
path, R13.3). Only the on-disk path changes.

### Taxonomy (per bounded context, e.g. `Catalog/`)

| Emitted file | Location |
| --- | --- |
| **Aggregate roots** | context root — `Catalog/Order.cs` (the aggregate is the entry point) |
| Nested entities (inside an aggregate) | `Catalog/Entities/` |
| Standalone entities | `Catalog/Entities/` |
| Value objects + generated `*Id` value objects | `Catalog/ValueObjects/` |
| Enums | `Catalog/Enums/` |
| Domain events | `Catalog/Events/` |
| Integration events | `Catalog/IntegrationEvents/` |
| Read models | `Catalog/ReadModels/` |
| Queries | `Catalog/Queries/` |
| Domain + application services | `Catalog/Services/` |
| Specifications | `Catalog/Specifications/` |
| Policies | `Catalog/Policies/` |
| Repository interfaces | `Catalog/Repositories/` |
| Unit of work, query handler seams, integration-event handlers (`IHandle*`), ACL translators | `Catalog/Abstractions/` |

### Cross-cutting

- **Runtime support** (`Koine/Runtime/*`) — unchanged. It is its own namespace
  and is not a bounded context, so kind subfolders do not apply.
- **Shared kernel** (R14.2) — the kind rules apply under the existing kernel
  folder. A kernel enum shared by Catalog and Ordering emits to
  `Catalog__Ordering/Kernel/Enums/Currency.cs`. The `__` kernel *name* is out of
  scope and unchanged; only flat-vs-kind layout changes.
- **Module path** (R13.3) — a type in module `Invoicing` of context `Billing`
  keeps namespace `Billing.Invoicing`; its kind folder is appended after the
  module path: `Billing/Invoicing/Entities/Invoice.cs`. "Context root" for an
  aggregate root generalizes to "root of its namespace folder", i.e.
  `Billing/Invoicing/Invoice.cs`.

## Design

The path is currently assembled at ~22 call sites as
`$"{FolderFor(ns)}/{Name}.cs"`. Each emit helper statically knows the kind of
file it produces, so the subfolder is decided at the call site — no runtime
classification needed.

Approach: add a kind segment to the path construction. Introduce a single helper

```csharp
// "" (root) for aggregate roots and runtime; otherwise the kind folder name.
private static string PathFor(string ns, string kindFolder, string fileName)
    => kindFolder.Length == 0
        ? $"{FolderFor(ns)}/{fileName}"
        : $"{FolderFor(ns)}/{kindFolder}/{fileName}";
```

and replace each `$"{FolderFor(ns)}/{x}.cs"` with the appropriate
`PathFor(ns, "<Kind>", "{x}.cs")`. `FolderFor` stays as the namespace→path
primitive.

### Call-site mapping

The kind is fixed per emit method, with one branch:

- `EmitValueObject` → `ValueObjects`
- `EmitEnum` → `Enums`
- `EmitEvent` → `Events`
- `EmitIntegrationEvent` → `IntegrationEvents`
- `EmitReadModel` → `ReadModels`
- `EmitQuery` → `Queries`
- `EmitSpecifications` → `Specifications`
- `EmitService` / `EmitApplicationService` → `Services`
- `EmitPolicy` → `Policies`
- `EmitIdValueObject` (unowned ids + entity-owned ids) → `ValueObjects`
- repository interfaces (`EmitUnitOfWork`'s repos / aggregate repo iface) →
  `Repositories`
- `EmitUnitOfWork`, `EmitQueryHandlerInterface` consumers, integration-event
  handler (`IHandle*`), ACL translator → `Abstractions`
- runtime emitters (`Koine/Runtime/*`) → `""` (root, unchanged)

**Branching case — `EmitEntityAndId`:**

- The entity file: `isRoot == true` → `""` (context root); `isRoot == false`
  (standalone entity or aggregate-nested entity) → `Entities`.
- Its generated `*Id`: always `ValueObjects`.

`EmitAggregate` emits the root via `EmitEntityAndId(isRoot: true)` and any nested
entities via `EmitEntityAndId(isRoot: false)`; the same branch covers both.

> Verify during implementation: confirm which emit method produces repository
> interfaces and whether the aggregate root iface vs. plain repo iface need
> different buckets. Both are interfaces; repository interfaces go to
> `Repositories`, the unit-of-work and other seams to `Abstractions`.

## Testing

- Existing tests in `tests/Koine.Compiler.Tests` assert generated file paths
  (e.g. snapshot/path assertions in the R-series tests). Update every path
  assertion to the new layout.
- Add focused assertions covering each new bucket: an aggregate root at context
  root, a value object under `ValueObjects/`, an enum under `Enums/`, a
  repository interface under `Repositories/`, a unit-of-work under
  `Abstractions/`, and a shared-kernel type under `<A>__<B>/Kernel/<Kind>/`.
- Determinism/idempotence guarantees are unaffected (paths are pure functions of
  the model); the byte-identical-re-emit test should still pass once snapshots
  are updated.
- Rebuild `demo/Shop.Domain` (which generates into `Generated/` during build) as
  an end-to-end check that the project still compiles with the new tree.

## Out of scope

- The `__` shared-kernel namespace/name convention (separate decision).
- Any namespace changes — namespaces are untouched.
- The glossary emitter and other targets.
- Folding kind into the namespace (rejected in favor of namespace == context).
