# Koine deferred language-design proposals

Generated by the design panel (2026-06-16). Each item: recommendation, breaking?, effort.

## Make `now` a first-class construct instead of a magic identifier

- **Recommendation:** adopt  |  **Breaking:** False  |  **Effort:** S  |  **Confidence:** high

### Proposal
PROBLEM TODAY
`now` is not a token and not registered in BuiltinOps. It is a plain `Identifier` (KoineLexer.g4:115) that flows in as an `IdentifierExpr`, and is special-cased by a hard-coded string literal "now" at four disconnected sites:
  1. src/Koine.Compiler/Ast/TypeResolver.cs:89  — `if (id.Name == "now") return Instant;`
  2. src/Koine.Compiler/Semantics/ExpressionChecker.cs:50 — `... && id.Name != "now"` (suppresses unknown-field)
  3. src/Koine.Compiler/Emit/CSharp/CSharpExpressionTranslator.cs:419 — `if (name == "now" && !_memberNames.Contains(name)) sb.Append("DateTimeOffset.UtcNow");`
  4. src/Koine.Compiler/Semantics/SemanticValidator.cs:689 — rejects `now` as a stored default (DiagnosticCodes.NowAsStoredDefault).
The string "now" is duplicated in 4 files with no single source of truth — exactly the anti-pattern BuiltinOps was created to prevent (its own doc-comment says "the language surface is defined in exactly one place").

DOES NAMING A FIELD `now` COLLIDE TODAY? No. All three resolution layers check member/scope membership BEFORE the `now` fallback (TypeResolver does `scope.TryGet` first; ExpressionChecker treats it valid because `scope.Contains` is already true; the emitter guards with `!_memberNames.Contains(name)`). The parser allows it: `softName : Identifier | ...` (KoineParser.g4:232) accepts `now` as a field name. So a field `now: Instant` is legal today and silently shadows the builtin. The only demo/example use is demo/Shop.Domain/Models/ordering.koi:106 (`submittedAt -> now`); no model declares a field named `now`.

RECOMMENDED DESIGN: a BuiltinOps registry entry + one helper, NOT a lexer keyword.
Keep `now` lexing as an `Identifier` (so it stays shadowable and is never a reserved word — no breaking change). Replace the four scattered string literals with a single registry:

  // BuiltinOps.cs
  /// Nullary builtins usable as a bare identifier in expressions. Each maps to a result type.
  public static readonly IReadOnlyDictionary<string,string> NullaryBuiltins =
      new Dictionary<string,string>(StringComparer.Ordinal) { ["now"] = "Instant" };
  public static bool IsNullaryBuiltin(string name) => NullaryBuiltins.ContainsKey(name);

Then:
  - TypeResolver.cs:89  ->  if (NullaryBuiltins.TryGetValue(id.Name, out var t)) return new TypeRef(t);
  - ExpressionChecker.cs:50  ->  ... && !BuiltinOps.IsNullaryBuiltin(id.Name) ...
  - SemanticValidator.cs:689  ->  unchanged in spirit, but compare against the registry (non-deterministic builtins) rather than the literal "now".
  - CSharpExpressionTranslator.cs:419  ->  keep `!_memberNames.Contains(name)` guard, but drive the C#-target mapping ("now" -> "DateTimeOffset.UtcNow") from a small target-side table keyed by the registry name (the registry itself stays target-agnostic, matching the existing IEmitter layering).

BEFORE (4 magic literals, no shared definition; a typo like "now " or "Now" silently becomes an unknown-field error with no hint).
AFTER (one registry; adding a future nullary builtin e.g. `today` or `newId` is a one-line dictionary entry consumed uniformly by resolver, checker, validator, and any emitter).

WHY NOT A LEXER KEYWORD (NOW token): it would (a) reserve `now`, breaking the currently-legal `now` field name and the soft-keyword philosophy the grammar deliberately follows (see KoineParser.g4:94-95, 235 declKeyword soft-keyword list), and (b) force `now` into `exprName`/`softName` alternatives anyway to stay usable after `.`. The grammar already proves the codebase prefers soft over hard keywords. So: registry yes, token no.

OPTIONAL DIAGNOSTIC IMPROVEMENT (cheap, additive): when a user declares a field named `now`, emit an INFO/WARN "field 'now' shadows the built-in current-instant value; reference it as a member to disambiguate" — surfaces the silent-shadowing behavior that exists today. This is the only place reasonable people might want a behavior change; recommend keeping it as a non-error warning to preserve compatibility.

### Grammar delta
None required (and none recommended). `now` remains the lexer `Identifier` rule (KoineLexer.g4:115); no new token, no change to softName/exprName/declKeyword (KoineParser.g4:232-235). Explicitly REJECT adding a `NOW` token: it would reserve the word and break the soft-keyword design. All changes are in the AST/Semantics/Emit C# layers, not the .g4 files.

### Migration impact
Zero .koi migration. No grammar or parser regen. Only 4 internal C# sites change (TypeResolver.cs:89, ExpressionChecker.cs:50, SemanticValidator.cs:689, CSharpExpressionTranslator.cs:419) plus a new ~6-line BuiltinOps entry. Demo/example impact: the single `now` usage at demo/Shop.Domain/Models/ordering.koi:106 continues to compile and emit `DateTimeOffset.UtcNow` byte-for-byte identically — verifiable by diffing generated output before/after. Docs that describe `now` (website/src/content/docs/reference/expressions.md:180-181, tutorials/commands-events-state.md:47/58/291, guides/roadmap.md:86, reference/commands-events-state.md:26) need NO content change since behavior is unchanged; optionally update expressions.md to mention `now` is a registered builtin and that a same-named field shadows it. If the optional shadow-warning is added, one new DiagnosticCodes entry + its golden-test snapshot would change.

### Rationale
Pure internal refactor that removes a 4-way-duplicated string literal and gives `now` a single source of truth in BuiltinOps, consistent with how every other builtin op is already defined and consumed by both the checker and emitters. It is non-breaking because `now` stays an Identifier (not reserved), so existing `submittedAt -> now` usage and the (currently legal, unused) ability to name a field `now` both keep working, with the existing member-shadowing precedence intact across all three layers. The registry approach is strictly more extensible than the status quo and cheaper than introducing a lexer token, which would be the only genuinely breaking option. Trade-off: it does not by itself add new capability (no new builtins shipped), so the immediate user-visible value is low — the payoff is maintainability and a clean seam for future nullary builtins (today/newId/etc.). The optional shadow-warning is the one place to be careful: keep it INFO/WARN, never an error, to stay non-breaking.

## Shared bare enum members impose ambiguity ceremony that surprises users (contextual typing by expected enum type)

- **Recommendation:** adopt  |  **Breaking:** False  |  **Effort:** S  |  **Confidence:** high

### Proposal
PROBLEM. Resolution of a bare enum member (e.g. `Cancelled` when both `OrderStatus` and `RefundStatus` declare it) is split across three layers:

- `ModelIndex.EnumsDeclaring(name)` -> all enums declaring the member; `Count > 1` == ambiguous.
- `ExpressionChecker` (src/Koine.Compiler/Semantics/ExpressionChecker.cs): resolves a bare member against the OTHER operand's concrete enum type (`ConcreteEnumType`/`ResolveEnumOperand`) for `== != < <= > >=`, and against the OTHER branch or the threaded `expected` type for `ConditionalExpr`/`CoalesceExpr`. `CheckEnumMemberResolvable` raises KOI0213 `AmbiguousEnumMember` only when nothing resolves it.
- The emitter (`CSharpExpressionTranslator.WriteIdentifier`, lines 425-440) is ALREADY more permissive: it qualifies a bare shared member using `enumHint ?? _expectedEnum`, falling back to `_enumMemberToType[name]` — it never fails.

So `expected` (the declared field/return/param enum type) is already plumbed into `ExpressionChecker.Check` and is consulted in the conditional/coalesce/transition/init/return/emit/default-initializer paths — but NOT in the comparison (`BinaryExpr`) path, where `CheckEnumMemberResolvable` only sees the sibling operand. That asymmetry is the surprise: the checker rejects a bare member as ambiguous even though (a) the expected enum type fully determines it and (b) the emitter would happily and correctly qualify it.

ROOT-CAUSE EXAMPLES (today's KOI0213, all unnecessary):
  // expected type = Bool, but both sides are the SAME intended enum
  // `Shared == Shared` where expected enum is known from surrounding default/return
  enum A { Shared, OnlyA }  enum B { Shared, OnlyB }
  // currently: KOI0213 on each `Shared` (no concrete operand, expected not consulted in comparison)

PROPOSED SEMANTIC RULE (single, additive): "A bare enum member resolves to the unique enum, among those declaring it, that the resolution CONTEXT pins. The context is, in priority order: (1) the other operand's concrete enum type, (2) the expected type flowing into the expression. KOI0213 is raised only when the member is declared by >=2 enums AND neither (1) nor (2) selects exactly one of them."

This is exactly the rule conditional/coalesce already follow and the rule the emitter already follows; the change makes the comparison path consistent with both.

CONCRETE GRAMMAR/LEXER/PARSER DELTA: none. Syntax is unchanged.

CODE DELTA (semantics only, ~6 lines in ExpressionChecker.cs):
- `CheckComparison(BinaryExpr b, TypeScope scope)` currently calls:
    CheckEnumMemberResolvable(b.Left, concreteRight, scope);
    CheckEnumMemberResolvable(b.Right, concreteLeft, scope);
  Change to pass the expected type as a fallback context (thread `expected` into CheckComparison, as ConditionalExpr/CoalesceExpr already receive it):
    CheckEnumMemberResolvable(b.Left, concreteRight ?? expected, scope);
    CheckEnumMemberResolvable(b.Right, concreteLeft ?? expected, scope);
  and likewise feed `concreteRight ?? expected` / `concreteLeft ?? expected` into the `ResolveEnumOperand(...)` calls so the inferred operand type used for the comparability check matches. `Check`'s `BinaryExpr` case (line 81-86) must forward `expected` to `CheckComparison`.

BEFORE (rejected today):
  value V { x: SomeEnum = Cancelled }            // OK already (default path, line 670)
  done: Bool = status == Cancelled               // OK already (concrete operand `status`)
  // but a comparison with no concrete operand and only an expected enum:
  pick: Bool = if flag then (Shared == Shared) else false   // KOI0213 x2  <-- surprise

AFTER (with the rule):
  - `status == Cancelled` still resolves via the concrete operand (unchanged).
  - A bare-vs-bare comparison resolves when the expected enum (from the enclosing default/return/transition/match context) declares the member uniquely; otherwise still KOI0213, now with the same "qualify as Enum.Member" message.

DIAGNOSTIC: keep KOI0213, but the message is already actionable ("ambiguous enum member 'X' (declared by A, B); qualify as Enum.X"). Optionally add the expected type to the hint when present: "...no enclosing type selects one; qualify as Enum.X". No new code needed.

OPTIONAL HARDENING (not required): also have `ExpressionChecker.Check`'s top-level `IdentifierExpr` case report KOI0213 for a bare member that is ambiguous AND unresolved by `expected`. Today a standalone bare ambiguous member in a non-comparison value position is silently emitted using `_enumMemberToType` (first owner) — a latent footgun. This is a separate, smaller correctness item; bundling it would make the change net-stricter in exactly one obscure spot, so I would do it as a follow-up to keep this change purely ceremony-reducing.

### Grammar delta
None. No lexer/parser/grammar changes. Resolution is entirely a semantic-analysis concern. Tokens involved are unchanged: bare members are plain Identifier; qualified members are MemberAccess (`EnumName.Member`); ASSIGN '=' (defaults), RARROW '->' (transitions), LARROW '<-' (factory init), ARROW '=>' (lambdas) are all unaffected.

### Migration impact
Purely additive: code that compiles today keeps compiling (the rule only ACCEPTS more inputs in the comparison path; a member resolvable under both operand and expected type still errors identically). Files that change: src/Koine.Compiler/Semantics/ExpressionChecker.cs (thread `expected` into CheckComparison; ~6 lines). Tests: existing R3DiagnosticsTests.Shared_enum_member_resolves_* keep passing; Ambiguous_bare_enum_member_is_reported (`Cancelled == Cancelled` in a `value V { flag: Bool = ... }` with expected=Bool) STILL reports KOI0213 because the expected type is Bool (not an enum) — so that negative test is unaffected. Ambiguous_enum_member_in_conditional_is_reported likewise unaffected (expected resolves to Bool via the `== ` result). Add 1-2 new positive tests for the newly-accepted case (bare-vs-bare comparison whose expected context pins one enum). Demo files (demo/Shop.Domain/Models/ordering.koi already uses `status == Cancelled` + explicit `RefundStatus.Cancelled`) are unchanged and still valid; no demo edits required. Docs: website/src/content/docs/reference/enums.md "Scoped (type-directed) member resolution" should add one sentence that the enclosing expected type is also a disambiguation source (it currently only mentions the field/operand), aligning the doc with the now-consistent behavior. Net affected demo files: 0; net affected doc files: 1 (one-sentence clarification).

### Rationale
This aligns three layers that already disagree: the emitter and the conditional/coalesce checker paths already use the expected enum type as a disambiguator; only the comparison path ignores it. The change removes a surprising KOI0213 in cases that the compiler can prove unambiguous and that the emitter would already lower correctly, so it strictly reduces required qualification without weakening safety — a member ambiguous under BOTH the operand AND the expected type still errors. It is the smallest possible change (pass `expected` as a fallback context, ~6 lines) and needs no grammar, parser, or codegen change. Trade-off: it makes the rule slightly less local ("the surrounding expected type can disambiguate"), but that is already the documented R3.5 mental model ("resolved against the field's enum type"), so it makes behavior match the docs rather than diverge. The one honest caveat is that contextual typing can let a typo in a bare member be silently absorbed by the expected enum; this is already true for defaults/transitions today, so the change does not introduce a new class of risk.

## Soft-keyword scheme: make the complexity cliff tractable via a dedicated reserved-position diagnostic (KOI0002) plus an authoritative docs/cheat-sheet, not a grammar change.

- **Recommendation:** adopt  |  **Breaking:** False  |  **Effort:** S  |  **Confidence:** high

### Proposal
## The soft-keyword set (from the grammar)

From `KoineParser.g4` lines 232-235, the grammar already classifies names into three tiers backed by `declKeyword`:

- **softName** = `Identifier | declKeyword | WHEN | IF | THEN | ELSE` — member names, lambda params, `.`-access names, transition/init/param/policy-arg targets.
- **exprName** = `Identifier | declKeyword | WHEN` — bare leading expression identifiers.
- **typeName** = `Identifier | declKeyword` — type references and context-map operands.

`declKeyword` (the soft set) is exactly: `context, value, quantity, entity, aggregate, enum, identified, by, root, command, requires, event, emit, states, create, spec, on, service, operation, policy, as, natural, sequence, guid, versioned, repository, operations, find, usecase, readmodel, from, query, import, module, acl, integration, publishes, subscribes, version`. Plus `when/if/then/else` admitted contextually via softName/exprName, and `since`/`deprecated` which are plain identifiers (annotation names, never tokenized as keywords — Parser.g4 line 95-96).

**Fully reserved (never identifiers):** `invariant` and `matches` (lexer pushes a regex mode on `matches`; `invariant` would be ambiguous against a member decl). Plus `true`/`false` (BoolLiteral) and the built-in generic type names `List/Set/Map/Range` (reserved by the semantic layer, KOI0908/KOI0107).

**Hard-`Identifier` positions** (Parser.g4 lines enumerated: 18, 76, 91, 104, 108, 111, 122, 125, 135, 139, 141, 153, 157, 172, 180, 190, 203, 206, 211, 213, 216) — the *declared name* of a context/module/service/operation/usecase/query/policy/value/quantity/entity/aggregate-root/command/factory/enum + its members/finder/event/integration-event, plus state-rule states, `emit`/policy event names, and repository `operations:` list. These require a bare `Identifier`.

## The actual cliff (and where it bites)

The docs (`reference/contexts-and-types.md` lines 195-227) already explain the rule well. The real gap is **diagnostics**: a user who writes `value from { ... }`, `command query(...) { }`, or `enum status { ... }` (using a soft keyword in a hard-Identifier position) gets a raw ANTLR error routed through `SyntaxErrorListener` (`Parsing/SyntaxErrorListener.cs`) as `KOI0001` with the message `mismatched input 'from' expecting Identifier`. That message neither names the rule nor tells them how to fix it. Symmetrically, `invariant: Int` (reserved word as a field) yields an equally opaque KOI0001.

## Proposal: KOI0002 "reserved-position" diagnostic (diagnostics-only, no grammar change)

1. **New code `KOI0002` in the Syntax band** (`DiagnosticCodes.cs`, `// ---- Syntax (KOI0001-0099)`): `ReservedWordInDeclarationName`.

2. **Post-process syntax errors in `SyntaxErrorListener`** (it already owns the offending IToken). When `offendingSymbol`'s token type is in the `declKeyword` set (or is `INVARIANT`/`MATCHES`) AND the ANTLR message contains `expecting Identifier`, replace the raw KOI0001 with a tailored KOI0002. Token-type membership is a cheap static `HashSet<int>` built from `KoineLexer` constants — no parser changes, no new tree-walking.

Before:
```
ordering.koi:7:8: error KOI0001: mismatched input 'from' expecting Identifier
```
After:
```
ordering.koi:7:8: error KOI0002: 'from' is a Koine keyword and cannot be used as a declaration
  name here. Keywords are usable as field, parameter, and member names, but the name that
  immediately follows 'value'/'entity'/'command'/etc. must be a plain identifier. Rename it
  (e.g. 'fromAddress') or, if you meant a field, move it inside the body.
```
For `invariant`/`matches`:
```
spec.koi:4:2: error KOI0002: 'invariant' is fully reserved in Koine and can never be a name
  (field, type, or otherwise). Choose a different name.
```

3. **LSP surfacing** — `KoineLanguageService` already classifies completions as `CompletionItemKind.Keyword`. Add a lightweight diagnostic provider path (or reuse the compiler's diagnostics) so KOI0002 shows as a squiggle in-editor with the same message; the message text is the load-bearing part and is shared from the compiler.

4. **Docs: promote to a first-class cheat-sheet.** The existing prose section is good but buried. Add a compact reference table to `reference/contexts-and-types.md` (and link it from `guides/editor-tooling.md`) with three columns: "soft (usable as field/param/member)", "hard-Identifier positions (declaration names)", "fully reserved (`invariant`, `matches`, `List/Set/Map/Range`, `true/false`)". Include the exact `declKeyword` list verbatim so it is authoritative rather than "...and more" (line 217 currently truncates). Add the KOI0002 message to `start/reading-the-output.md` so the error is searchable.

## Why diagnostics over grammar

The soft-keyword design is deliberate and valuable (DDD vocabulary like `value`, `from`, `query`, `service`, `policy` are also natural field names). Hardening the grammar (reserving words) would break existing models and remove the feature's whole point. The cliff is purely a *legibility* problem at the failure boundary — solved by a targeted message, not by changing what parses.

### Grammar delta
None required. The proposal is intentionally grammar-free. The existing rules (KoineParser.g4 lines 232-235: softName / exprName / typeName / declKeyword) already encode the soft set and are the source of truth for the diagnostic's token list. The only code touched: DiagnosticCodes.cs (add KOI0002 ReservedWordInDeclarationName in the 0001-0099 band) and Parsing/SyntaxErrorListener.cs (inspect offendingSymbol.Type against a static HashSet built from KoineLexer token constants for the declKeyword set plus INVARIANT/MATCHES; when the ANTLR message indicates 'expecting Identifier', emit KOI0002 with a tailored message instead of KOI0001). Optionally route the same message into the LSP diagnostic path in KoineLanguageService.

### Migration impact
No existing .koi files break — all demo/example models (demo/Shop.Domain/Models/*.koi, examples/*.koi) already parse and none use a keyword in a hard-Identifier position, so KOI0002 never fires on valid code. The change only alters the *text and code* of an error that is already a hard syntax failure (previously KOI0001). Migration is additive: (1) tests asserting the literal 'KOI0001' / 'mismatched input ... expecting Identifier' text for keyword-in-name cases would need updating to KOI0002 — grep shows the relevant test files are R3DiagnosticsTests.cs, R13ModulesImportsTests.cs, R14ContextMapsTests.cs; add a focused KOI0002 test. (2) Docs: edit reference/contexts-and-types.md (replace the truncated 'and more' list with the full verbatim declKeyword set + a 3-column table), link from guides/editor-tooling.md, and add the KOI0002 message to start/reading-the-output.md. No generated C# output changes; no grammar regeneration.

### Rationale
The grammar's soft-keyword machinery (softName/exprName/typeName/declKeyword) is sound and the docs already describe it; the only real friction is that hitting a hard-Identifier position produces a generic ANTLR KOI0001 ("mismatched input 'from' expecting Identifier") that never mentions the soft/hard-keyword concept or how to recover. A single, tailored KOI0002 emitted from the existing SyntaxErrorListener (which already has the offending token) closes the gap with zero grammar risk. Trade-off: the heuristic keys on the ANTLR phrase "expecting Identifier" plus the offending token type, so it is coupled to ANTLR's message wording for the trigger condition (the message we emit is fully ours). That coupling is acceptable and guarded by tests. Promoting the docs section to a complete, verbatim cheat-sheet (the current list ends in "and more") removes ambiguity about which words are soft vs reserved.

## Hyphenated context-map roles (shared-kernel, customer-supplier, open-host, anti-corruption-layer, published-language) vs. the no-hyphen identifier rule

- **Recommendation:** adopt  |  **Breaking:** False  |  **Effort:** S  |  **Confidence:** high

### Proposal
PROBLEM (verified): The lexer's general identifier rule disallows hyphens — `Identifier : [a-zA-Z_] [a-zA-Z0-9_]*` (KoineLexer.g4:115) — and `-` is exclusively MINUS (KoineLexer.g4:102). Yet five context-map roles are hard-coded as hyphenated single-literal tokens (KoineLexer.g4:53-61): SHARED_KERNEL 'shared-kernel', CUSTOMER_SUPPLIER 'customer-supplier', ANTI_CORRUPTION 'anti-corruption-layer', OPEN_HOST 'open-host', PUBLISHED_LANGUAGE 'published-language'. The grammar even carries a defensive comment: "hyphenated roles MUST be single literal tokens; '-' is MINUS". So the hyphens are not a lexing accident — they are a deliberate fixed-keyword workaround. The inconsistency is purely cosmetic/conceptual, not a parsing bug: these literals never collide with anything because they only appear in ONE position — `relationRole` after `COLON` in `relationDecl` (KoineParser.g4:42-53) — and are never usable as identifiers.

RECOMMENDATION: KEEP the hyphens, but DOCUMENT them as an explicit, named token class rather than leaving them as an undocumented special case. This is the lowest-risk option and preserves the strongest argument the design has: these are the literal canonical names from Evans' DDD (`shared-kernel`, `anti-corruption-layer`, `open-host`), and matching the literature verbatim is more valuable than lexer-internal uniformity. Concretely:

1. Add a grammar-level fragment + comment block that names this class, e.g. a `// ---- Context-map role keywords (hyphenated contextual keywords) ----` section header, so a reader immediately sees these are an intentional class, not stray tokens. (No token-rule change.)
2. In KoineParser.g4 keep the `relationRole` alternative list as the single authoritative enumeration (it already is) and add a one-line comment that hyphens are legal ONLY here.
3. Document the rule explicitly in website/src/content/docs/reference/context-maps-integration.md — it already half-says this at line 42-43 ("The role names are single hyphenated tokens too"). Promote that into a named concept: "Contextual keywords: hyphens are part of the spelling of context-map role names and appear nowhere else in the language."

BEFORE (today, undocumented special case):
```
// R14 — context map (hyphenated roles MUST be single literal tokens; '-' is MINUS).
SHARED_KERNEL      : 'shared-kernel' ;
CUSTOMER_SUPPLIER  : 'customer-supplier' ;
ANTI_CORRUPTION    : 'anti-corruption-layer' ;
```
AFTER (documented token class, behavior identical):
```
// ---- Context-map role keywords (R14) ---------------------------------------
// These are CONTEXTUAL keywords: the hyphen is part of the spelling and is the
// ONLY place '-' appears outside the MINUS operator. Each is a single literal
// token (so maximal munch never splits 'shared-kernel' into IDENT MINUS IDENT)
// and is valid solely as `relationRole` after the ':' in a relationDecl.
SHARED_KERNEL      : 'shared-kernel' ;
CUSTOMER_SUPPLIER  : 'customer-supplier' ;
ANTI_CORRUPTION    : 'anti-corruption-layer' ;
```

ALTERNATIVE CONSIDERED — switch to a hyphen-free convention (e.g. snake_case `shared_kernel` or PascalCase `SharedKernel`). This would make roles obey the Identifier rule and remove the special token class entirely. REJECTED as the primary path because: (a) it breaks every existing model and the docs/glossary for no functional gain; (b) it discards the verbatim-DDD-vocabulary property, which is the feature's selling point; (c) `SharedKernel`/`shared_kernel` reads worse than the canonical literature term. If the team nonetheless wants uniformity, the migration is mechanical and the design supports it cleanly because roles live in exactly one grammar position (see migrationImpact) — that path is "adopt-with-migration", offered as a fallback only.

A note on a third spelling some DSLs use — quoted-string roles (`: "shared-kernel"`). REJECTED: it makes the most-used vocabulary in the strategic map look like data rather than language, regressing readability.

### Grammar delta
KEEP path (recommended): no rule changes. Add a labeled comment section in KoineLexer.g4 around lines 53-61 documenting SHARED_KERNEL/CUSTOMER_SUPPLIER/CONFORMIST/ANTI_CORRUPTION/OPEN_HOST/PUBLISHED_LANGUAGE as contextual keywords; add a one-line note on KoineParser.g4 relationRole (line 47). Token IDs, parser rules, and KoineModelBuilderVisitor.BuildRelationKind (KoineModelBuilderVisitor.cs:138-147) all unchanged.

MIGRATION path (fallback only, if convention change is chosen): edit the five token literals in KoineLexer.g4:56-61 (e.g. 'shared_kernel', 'customer_supplier', 'anti_corruption_layer', 'open_host', 'published_language'); the MINUS-collision comment at line 53 becomes removable and 'anti-corruption-layer' no longer needs to win maximal munch. Parser rule names, token names, and the visitor stay identical (they key on token presence, not text). Update SemanticValidator.RoleName (SemanticValidator.cs:279-288) which reproduces the canonical wire spellings used in diagnostic messages.

### Migration impact
KEEP path: zero migration. No .koi file, doc, or test changes. Only additive comments.

MIGRATION path (if convention change adopted): the actual hyphenated ROLE KEYWORDS occur in only ONE model file — demo/Shop.Domain/Models/context-map.koi (7 lines: shared-kernel, conformist, customer-supplier, open-host x2, partnership, anti-corruption-layer). All other .koi hits (catalog/ordering/shipping/customers/payments.koi) are in COMMENTS only and need no functional change. Tooling: 1 line in tooling/koine-textmate/syntaxes/koine.tmLanguage.json:65 (highlighter regex). Tests: ~65 string occurrences across tests/Koine.Compiler.Tests/{R14ContextMapsTests, R14SharedKernelAclTests, R14IntegrationEventsTests, R15CheckTests, EmitterSnapshotTests}.cs use the hyphenated spellings in inline fixtures and message asserts. Docs: ~14 files under website/src/content/docs and demo/reference reference the spellings. So a convention switch is mechanical but touches ~1 model + 1 tooling file + 5 test files + ~14 docs, and would break any external user models — hence 'breaking' for that path. The recommended KEEP path is non-breaking, which is why `breaking=false`.

### Rationale
The hyphens are an intentional design choice (verbatim Evans DDD vocabulary), not a lexer mistake. They cause zero parsing ambiguity because roles occupy a single, post-COLON grammar slot in relationRole (KoineParser.g4:42-53) and are never identifiers — so the apparent inconsistency with the no-hyphen Identifier rule is conceptual, not technical. The cheapest fix that resolves the reviewer's concern is to make the special case explicit and documented (a named contextual-keyword class) rather than to churn the surface syntax. Switching conventions would break every model/doc/test and trade away the feature's main value (matching the literature) for internal tidiness no user benefits from. Trade-off: 'keep + document' leaves one genuinely special token class in the lexer, but it is now labeled and bounded, which is exactly what a reader needs.

## Expression sublanguage has no local bindings, forcing redundant derived fields

- **Recommendation:** adopt-with-migration  |  **Breaking:** False  |  **Effort:** M  |  **Confidence:** high

### Proposal
Add a `let`-binding form at the TOP of the expression grammar so any derived computation can be named and reused inside one expression without promoting it to a public field.

SYNTAX (chosen form): `let NAME = expr (, NAME = expr)* in body`
- Sequential bindings (each binding sees the previous ones), terminated by `in`.
- The whole `let ... in body` is itself an expression, so it nests anywhere a value is expected: invariants, derived-field defaults (`= ...`), `requires`, factory `<-` inits, spec bodies, conditional/coalesce branches.

GRAMMAR DELTA (additive only):
- Lexer (KoineLexer.g4): two new keyword tokens
    LET : 'let' ;
    IN  : 'in' ;
  Add LET and IN to the `expression-position` keyword fallbacks so existing models that used "let"/"in" as field names keep working (see migration note). Place them anywhere among the keyword block; no maximal-munch conflicts (`let`/`in` don't prefix any existing token).
- Parser (KoineParser.g4): make `let` the new lowest-precedence layer, above the guard:
    expression : letExpr ;
    letExpr    : LET letBinding ( COMMA letBinding )* IN letExpr   // let x = e, y = e in body
               | guardExpr ;
    letBinding : softName ASSIGN expression ;
  (Everything else in the precedence chain is unchanged. `guardExpr` remains the fall-through, so all existing expressions parse identically.)
- Soft-keyword tables: add LET and IN to `declKeyword` (so they remain legal as TYPE/MEMBER names and after `.`). The binding name uses `softName`, so a binding may shadow a keyword-named member, consistent with lambda params.

AST DELTA (Ast/Expressions.cs): one new node
    public sealed record LetExpr(IReadOnlyList<LetBinding> Bindings, Expr Body) : Expr;
    public sealed record LetBinding(string Name, Expr Value);
Model builder (KoineModelBuilderVisitor): new `BuildLet` invoked from `BuildExpression`; builds each binding's value via `BuildExpression`, then the body.

TYPE RESOLVER / CHECKER DELTA:
- TypeResolver.Infer: add `case LetExpr l:` — fold bindings into the scope (`scope = scope.With(b.Name, Infer(b.Value, scope))` for each, in order), then `Infer(l.Body, scope)`. The existing `TypeScope.With` already supports this; no new infra.
- ExpressionChecker.Check: add `case LetExpr` — check each binding value in the accumulating scope, then check the body in the extended scope. Bindings participate in unknown-field/enum/optional diagnostics for free because they live in the normal scope.

LOWERING TO C# (CSharpExpressionTranslator):
Bindings lower to a C# block-bodied lambda invoked inline (an IIFE), which keeps `Translate` returning a single C# expression string (the translator's whole contract):
    ((Func<T>)(() => { var x = <expr>; var y = <expr>; return <body>; }))()
But since Koine never needs the explicit delegate type, the idiomatic and simplest lowering is a local-function-free LINQ-style `let` is NOT available in expression position, so use the IIFE WITHOUT a named type via a target-typed form is not possible at expression level. Concrete, compilable choice: render as a parenthesized invocation of a lambda with `var` locals:
    (new Func<RET>(() => { var __let_x = EXPR; var __let_y = EXPR; return BODY; }))()
where RET is the inferred body type rendered through the existing type renderer. To avoid needing RET, the recommended emit is even simpler — hoist the bindings as `var` statements is NOT possible in a pure expression slot, so the IIFE is the mechanism. Register each binding name as a local via the existing `PushLocal(name, inferredType)` / `PopLocal` pair (already used for lambda/command params) so the body renders the binding references verbatim (camelCase) exactly like a lambda parameter — no new identifier path needed.

BEFORE (demo/Shop.Domain/Models/ordering.koi, OrderLine) — `lineTotal` exists ONLY so `payable` can reuse it; it leaks as a public derived property:
    lineTotal: Money = unitPrice * quantity
    payable:   Money = if quantity >= 10 then lineTotal * 0.9 else lineTotal

AFTER — one private computation, one public field; `lineTotal` no longer pollutes the type's surface:
    payable: Money =
      let lineTotal = unitPrice * quantity
      in if quantity >= 10 then lineTotal * 0.9 else lineTotal

Lowers to (Property mode):
    public Money Payable =>
      (new Func<Money>(() => { var lineTotal = UnitPrice * Quantity; return Quantity >= 10 ? lineTotal * 0.9m : lineTotal; }))();

INVARIANT example (reuse a subtotal in two clauses without a field):
    invariant
      let charged = lines.sum(l => l.payable)
      in charged.amount >= 0 && charged.amount <= creditLimit
    "an order cannot exceed the customer credit limit"

### Grammar delta
Lexer (KoineLexer.g4): add tokens `LET : 'let' ;` and `IN : 'in' ;`. No maximal-munch ordering concern (neither prefixes an existing literal). Parser (KoineParser.g4): insert a new lowest layer — change `expression : guardExpr ;` to `expression : letExpr ;`, then add `letExpr : LET letBinding (COMMA letBinding)* IN letExpr | guardExpr ;` and `letBinding : softName ASSIGN letExpr ;` (use letExpr/expression for binding values so nested lets work). Add `LET` and `IN` to the `declKeyword` rule (keeps them usable as type/member names and after `.`); binding names already use `softName`. No changes to any other expression rule — all existing inputs reduce through the unchanged `guardExpr` alternative.

### Migration impact
Purely additive at the grammar level: every existing expression still parses via the unchanged `guardExpr` fall-through, so no .koi file needs to change to keep compiling. The ONLY breakage risk is a model that used `let` or `in` as a bare leading expression identifier (an `exprName`); a repo grep of demo/Shop.Domain/Models/*.koi and examples/*.koi shows `let`/`in` appear only inside comments, never as identifiers — so zero demo files break in practice. Mitigation already built in: adding LET/IN to `declKeyword` (the soft-keyword set) keeps them legal as MEMBER names, `.`-accessed names, and TYPE names; they only become reserved as a LEADING expression token, mirroring how `if`/`then`/`else` are already treated. Files that could OPTIONALLY adopt the feature to drop a redundant field: ordering.koi (`lineTotal`→local inside `payable`), and similar simplifications possible in examples/billing.koi (`subtotal`) and customers.koi. New compiler code: 1 AST record pair (Expressions.cs), 1 `BuildLet` in the model builder, 1 `case LetExpr` each in TypeResolver.Infer and ExpressionChecker.Check, and 1 `case LetExpr` in CSharpExpressionTranslator reusing existing PushLocal/PopLocal. Docs: add a short section to website feature-catalogue/expression docs; no existing doc statement becomes false.

### Rationale
This is a real, demonstrated pain point: `ordering.koi` already carries `lineTotal` as a public derived property whose sole reason to exist is reuse inside `payable`. That over-exposes the model's surface and emits a property nobody should depend on. A `let` binding makes the intermediate genuinely private (a C# local), keeping the public DDD surface clean — the whole point of the language.

Why `let ... in` over a `where` clause: (1) it is an EXPRESSION, so it composes uniformly in every expression position (invariant, default, requires, factory init, conditional branch) with zero new grammar attach-points; a `where` clause would have to be bolted onto each declaration form (member, invariant, requires, initialization) separately — far more grammar surface and more breaking risk. (2) Sequential, prefix bindings read top-down and match the existing functional, statement-free feel of the sublanguage. (3) The lowering reuses the EXISTING `PushLocal`/`PopLocal` machinery — binding references render exactly like lambda parameters, so no new identifier-resolution path is needed in the translator, checker, or resolver; each just adds one `case`.

Trade-offs: (a) the IIFE lowering (`new Func<T>(() => {...})()`) is heavier than a hand-written local and needs the body's inferred type to name the delegate; for un-inferable bodies the emitter would fall back to wrapping or refuse — acceptable since the checker already infers these types for diagnostics. A future refinement could hoist to a private computed helper method when the binding sits at field level, producing cleaner code, but the IIFE keeps the translator's single-expression contract intact for v1. (b) `let`/`in` become reserved in expression position.

## Command bodies cannot compute or return values; only set-field/requires/emit

- **Recommendation:** adopt-with-migration  |  **Breaking:** False  |  **Effort:** M  |  **Confidence:** high

### Proposal
PROBLEM TODAY
A `commandDecl` (KoineParser.g4:157) has no return-type slot and `commandStmt` (lines 163-166) is exactly one of requires | transition | emit. `WriteCommand` (CSharpEmitter.cs:754-843) therefore hard-codes `public void <Name>(...)`. There is no way for a command to hand a value back to the caller — even the most common DDD case, "command creates a child entity / generates a token / returns a receipt id". By contrast factories already return (`public static <Entity> Open(...)` with `return instance;`, WriteFactory CSharpEmitter.cs:986-998) and use-cases already accept an optional return type (`usecaseDecl ... (COLON typeRef)?` -> `Task<T>` vs bare `Task`, application-layer.md:148). Commands are the odd one out.

PROPOSAL — opt-in return on commands via a typed signature + a `result` clause
Two coordinated additions, both fully backward compatible (everything is optional):

1) Grammar — optional return type on the command signature, mirroring usecaseDecl exactly:
   BEFORE: commandDecl : COMMAND Identifier ( LPAREN paramList? RPAREN )? LBRACE commandStmt* RBRACE ;
   AFTER:  commandDecl : COMMAND Identifier ( LPAREN paramList? RPAREN )? ( COLON typeRef )? LBRACE commandStmt* RBRACE ;

2) Grammar — a new `result` clause as a fourth commandStmt alternative (reuse RESULT keyword + existing ASSIGN '=' token, NOT a new arrow — keeps the arrow taxonomy intact: -> stays transition-only):
   commandStmt   : requiresClause | transition | emitClause | resultClause ;
   resultClause  : RESULT expression ;     // `result tokenId`  (RESULT 'result' added to lexer)
   I chose a bare `result <expr>` over `result = <expr>` because requires/transition/emit are all keyword-led, arrow-free statements; `result newToken` reads consistently with `emit X`. (`=` is available as a fallback if reviewers prefer `result = newToken`.)

SEMANTICS / EMIT MAPPING (WriteCommand)
- No return type declared: unchanged. `public void Submit()`. A `result` clause without a declared type is a semantic error (new diagnostic, see grammarDelta).
- Return type declared: `public <MappedType> <Name>(...)`. The method body keeps its fixed ordering (requires -> transitions -> CheckInvariants -> emits) and then appends `return <translated result expr>;` as the LAST statement, so the value is computed over the post-mutation state and only reached if all guards + the invariant re-check pass. The result expression is translated in the same scope as emit payloads (parameters still pushed as locals, members as properties), so it can reference parameters, `id`, and just-assigned fields.
- Exactly one `result` required when a type is declared; zero or >1 is a diagnostic.

BEFORE/AFTER EXAMPLE (entity command)
  // ordering.koi — issue a refund voucher and hand its id back
  command refund(amount: Money): VoucherId {
    requires status == Paid "only a paid order can be refunded"
    status -> Refunded
    emit OrderRefunded(orderId: id, amount: amount)
    result VoucherId.New()
  }
Emits:
  public VoucherId Refund(Money amount)
  {
      if (!(Status == OrderStatus.Paid))
          throw new DomainInvariantViolationException(type: nameof(Order), rule: "only a paid order can be refunded");
      // legal-transition guard...
      Status = OrderStatus.Refunded;
      CheckInvariants();
      var __result = VoucherId.New();          // computed once, before events reference it
      _domainEvents.Add(new OrderRefunded(Id, amount));
      return __result;
  }
(If the result expr is referenced by an emit payload, hoist it to a `var __result` before the emits as shown; otherwise inline `return <expr>;`.)

INTERACTION WITH EVENTS
- Ordering is preserved: events are still recorded after the invariant re-check; the returned value is computed at the same scope but the `return` is the terminal statement. This keeps the existing guarantee "an invalid post-state throws before any event is recorded" (CSharpEmitter.cs:818-840) AND adds "a value is only returned from a fully-valid, fully-eventful aggregate."
- A common idiom — returning the id/value that an emitted event also carries — is naturally expressible: compute it once into `__result`, pass it into the emit payload, and return it (single source of truth, no recomputation).
- Does NOT violate CQS in a way that surprises: this is the standard "command returns the identity of what it created" exception widely accepted in DDD (same rationale factories already use by returning the aggregate).

USE-CASES
No change needed — `usecaseDecl` already supports the optional return type and maps to `Task<T>` (application-layer.md:148). This proposal simply makes entity/aggregate commands consistent with use-cases and factories. Optionally, document that a use-case orchestrating a value-returning command threads that command's result into its `Task<T>`.

AST CHANGES
- CommandDecl (Nodes.cs:313-316): add `TypeRef? ReturnType`.
- New `public sealed record ResultClause(Expr Value) : CommandStmt;` (Nodes.cs ~348).
- KoineModelBuilderVisitor.BuildCommand (line 400-412): read optional `ctx.typeRef()`; BuildCommandStmt (457-481): add the resultClause branch.

### Grammar delta
Lexer (KoineLexer.g4, near line 26): add RESULT : 'result'; (place before/with other command keywords; no maximal-munch conflict — 'result' is unique). Parser (KoineParser.g4): (a) commandDecl line 157 — insert ( COLON typeRef )? before LBRACE; (b) commandStmt lines 163-166 — add `| resultClause`; (c) add `resultClause : RESULT expression ;`. AST (Nodes.cs): CommandDecl gains `TypeRef? ReturnType`; add `ResultClause(Expr Value) : CommandStmt`. Visitor (KoineModelBuilderVisitor.cs): BuildCommand reads ctx.typeRef(); BuildCommandStmt handles ctx.resultClause(). Emitter (CSharpEmitter.cs WriteCommand ~754): method return type from cmd.ReturnType (default void); collect resultClause; emit `return <expr>;` (or hoist to `var __result` when an emit payload references it) as final statement. New semantic diagnostics: KOI09xx 'result clause requires a declared command return type', KOI09xx 'command with a return type must have exactly one result clause', and a type-match check between the result expression and the declared return type (reuse existing expression-type checking used for operation bodies / derived members).

### Migration impact
Zero forced migration. Both additions are optional: the new `( COLON typeRef )?` and the new resultClause alternative do not affect any existing parse. All 7 existing commands across demo/Shop.Domain/Models/{ordering,payments,shipping}.koi remain valid and emit byte-identical `public void` methods (verified: none declares a return type). No existing .koi file changes. Docs to update: commands-events-state.md (add a 'returning a value' subsection + the emitted-method example), feature-catalogue.md (command row, mirroring the factory 'return' note), and the LSP/grammar tooling (R17) for the new `result` keyword and command-return-type completion. Tests: add golden-file cases for (a) command with return type + result, (b) result referenced by an emit payload (hoist path), (c) the three new diagnostics. EBNF/syntax-highlight grammar shipped for editors needs the RESULT keyword added.

### Rationale
It removes a real, surprising asymmetry: factories return, use-cases return, only commands cannot — yet "a command returns the id of what it created" is a mainstream DDD pattern. The design reuses existing, proven seams: the optional `: typeRef` is copied verbatim from usecaseDecl, the body-ordering and scope-management machinery in WriteCommand already exists, and the `return` epilogue mirrors WriteFactory's `return instance;`. Reusing RESULT+expression (no new arrow) keeps the arrow taxonomy clean (-> stays transition-only, <- factory-only), so it does not muddy the language's distinctive grammar. Trade-offs: (1) it introduces an explicit CQS exception — mitigated by making it opt-in and documenting it as the create-and-return-id idiom; (2) `result` lets a command compute an expression, a small expansion of command-body power, but it is a single terminal value, not arbitrary logic, so commands stay declarative. Net: low-risk, high-consistency, fills a genuine expressiveness gap.

## Unify/clarify the three assignment-like arrows (-> vs <- vs =)

- **Recommendation:** adopt-with-migration  |  **Breaking:** True  |  **Effort:** S  |  **Confidence:** high

### Proposal
RECOMMENDED: collapse LARROW `<-` into RARROW `->`, keep `=` and `->` only. This drops one of the three arrows and removes the rarest, least-pedagogically-justified one.

AUDIT (the load-bearing facts):
- `<-` (LARROW) has EXACTLY ONE grammar use: `initialization: softName LARROW expression` (factory field init). It appears in ZERO demo/example .koi files (verified across demo/Shop.Domain/Models/*.koi, examples/*.koi, examples/versioning/**). Its only real exercise is tests/Koine.Compiler.Tests/R8FactoryTests.cs and one demo/README.md line. So the "newcomer must learn three arrows" cost is paid for a symbol that newcomers literally never see in the showcase.
- `->` (RARROW) carries THREE jobs already and readers tolerate it fine: state-rule edges (stateRule), command transitions (transition: `status -> Submitted`), and ACL/context-map direction (aclMapping, relationArrow). The unifying gloss for all three is identical: "left becomes / flows to right."
- `=` is structurally different: it appears at DECLARATION sites (member defaults, spec/operation/readmodel bodies) and is the equality operator `==`. It is never a statement; it is part of a declaration `name: T = expr`.

THE MENTAL MODEL (this is what the docs should teach, and it justifies the merge):
  `=`  = "this field's definition / default, evaluated once at the declaration." (declaration-time, structural)
  `->` = "an effect on the aggregate's state: left-hand field takes on right-hand value." (statement-time, behavioral)
  `<-` adds NO new axis — initialization IS just "the field takes on this value," identical in meaning to a transition. The only difference is timing (during create vs during a command), which the enclosing `create {}` vs `command {}` block ALREADY makes unambiguous. The arrow direction is redundant with the block keyword.

BEFORE:
  command record(amount: Decimal) { balance -> amount }
  create open(c: CustomerId) { customer <- c   lines <- lines }

AFTER:
  command record(amount: Decimal) { balance -> amount }
  create open(c: CustomerId) { customer -> c   lines -> lines }

Grammar delta: change `initialization: softName LARROW expression` to use RARROW (or merge `transition`/`initialization` into one `assignmentStmt: softName RARROW expression` referenced by both commandStmt and factoryStmt). The two are already validated by separate visitors (SemanticValidator: transition target must be a non-derived field; initialization additionally tracks per-field once-only + uninitialized-member detection R8.2) — those checks key off the enclosing block, NOT the token, so they are unaffected. LARROW token can be deleted from the lexer; BIARROW `<->` no longer needs the "before LARROW" maximal-munch ordering note but the ordering before LT is still required.

ALTERNATIVE (option a, if zero breakage is mandatory): keep all three but make `=` vs `->` the ONLY conceptual split taught (declaration default vs runtime effect), and explicitly document `<-` as "the create-block spelling of `->`, oriented to read 'field receives value'." Add an LSP hover + a diagnostic that, on `<-` inside a `command` (or `->` inside a `create`), suggests the correct arrow with the rationale. This keeps coherence without migration but retains the learning cost.

### Grammar delta
KoineParser.g4: replace `initialization: softName LARROW expression` with RARROW; ideally introduce `assignmentStmt: softName RARROW expression` and reference it from both commandStmt and factoryStmt (keeping distinct AST node types Transition/Initialization via parser context or a post-parse split so existing SemanticValidator visitors and R8.2 once-only/uninitialized checks keep working unchanged). KoineLexer.g4: delete `LARROW: '<-'` (line 100) and its maximal-munch comment; remove the now-stale 'before LARROW' note on BIARROW (line 99) — BIARROW must still precede LT. No change to ASSIGN or RARROW.

### Migration impact
Demo/examples: ZERO .koi files use `<-`, so the entire demo/ and examples/ trees compile unchanged — the showcase is unaffected. Tests: R8FactoryTests.cs has ~10 `<-` occurrences that must be rewritten to `->` (mechanical sed `s/ <- / -> /`). Docs: demo/README.md line 79 (`field <- value`) and the website Starlight pages (feature-catalogue, architecture, commands-events-state, etc.) that describe factory init need a one-line update plus the new two-arrow mental-model section. Tooling: KoineFormatter is whitespace/token-passthrough (no per-arrow logic found), so no formatter change; LSP grammar/highlighting must drop the `<-` token. A trivial codemod (regex over `create {}` blocks) can migrate any external .koi files, but since `->` was previously illegal inside `create`, a blind global `<- -> ->` is safe. Net: handful of test lines + doc edits; no model behavior change.

### Rationale
First-principles: a token should earn its place by encoding a distinction the reader cannot get elsewhere. `<-` fails this test — its sole distinction (init vs mutate) is fully implied by the enclosing `create {}` vs `command {}` keyword, so the arrow direction is pure redundancy that newcomers must still decode. Merging into `->` yields a clean two-arrow story (`=` = declaration default; `->` = state effect) that maps 1:1 onto how every demo already reads. The trade-off is mild: some find `<-` evocative ('flows into the new object'), and the merge is technically breaking. But the breakage surface is tiny (see migrationImpact) and the payoff — one fewer symbol on the cheat-sheet, and a teachable model where the remaining two arrows are genuinely orthogonal — is real. I rank this above option (a) because option (a) preserves the cost it set out to remove; the audit shows the symbol isn't pulling its weight.

## Emit invariant-free, derived-field-free value objects as C# records instead of ValueObject-derived classes

- **Recommendation:** defer  |  **Breaking:** True  |  **Effort:** M  |  **Confidence:** high

### Proposal
SCOPE OF THE ELIGIBLE SUBSET
The proposal only touches VOs that have ALL of: no invariants, no derived/computed members, not a quantity, and no emitted arithmetic operators (i.e. not in _scalarNeeds/_additiveNeeds). Surveying every model:
- Eligible (3): PaymentReceipt (payments.koi), GatewayResult (legacy.koi), SalePeriod (catalog.koi).
- NOT eligible: Money/Email/Price/Sku (invariants), Weight (quantity), PostalAddress (derived `formatted`), OrderLine (derived `subtotal` + is a scalar-operator target).
So 3 of ~12 VOs would change shape.

DOES A RECORD SATISFY THE TEST? Partially, by accident of coverage, not by intent.
The test (tests/Koine.Compiler.Tests/GeneratedCodeTests.cs:75 Value_object_is_a_class_deriving_ValueObject_not_a_record) asserts on Billing.Money ONLY:
  Assert.Equal("ValueObject", money.BaseType!.Name);
  Assert.False(money.IsSealed && money.GetMethods().Any(m => m.Name == "<Clone>$"), ...);
Money has an invariant (`amount >= 0`), so under the proposal Money stays a hand-written ValueObject class. The test therefore still passes. Reflection-wise a `record class` IS a class (Type.IsClass == true), but a record does NOT derive ValueObject (its BaseType is `object`) AND it emits the `<Clone>$` method — so if the test were ever pointed at an eligible VO it would FAIL on both assertions. The test name encodes a global intent ("value objects must not be records") that the proposal violates for the eligible subset; passing is purely because the fixture's tested VO is ineligible.

BEFORE (current, all VOs):
  public sealed class PaymentReceipt : ValueObject {
      public string Reference { get; }
      public decimal Amount { get; }
      public PaymentReceipt(string reference, decimal amount) { Reference = reference; Amount = amount; }
      public override string ToString() => $"PaymentReceipt {{ Reference = {Reference}, Amount = {Amount} }}";
      protected override IEnumerable<object?> GetEqualityComponents() { yield return Reference; yield return Amount; }
  }

AFTER (proposal, eligible only):
  public sealed record PaymentReceipt(string Reference, decimal Amount);

PUBLIC API DELTA for the eligible subset:
- Gains: `with` expressions (`receipt with { Amount = 5m }`); positional deconstruction; compiler-generated value ToString (`PaymentReceipt { Reference = ..., Amount = ... }` — nearly identical to current).
- Loses: derivation from `ValueObject`, so it no longer participates in the base `operator ==(ValueObject?, ValueObject?)` / polymorphic-by-base equality, and no longer flows through GetEqualityComponents. Record gets its own typed `operator ==`, `Equals`, `GetHashCode`, `<Clone>$`.
- Constructor changes from a body-block ctor to a primary ctor; param defaults (`= null`, enum-default coalescing) and `Ordered()/Unordered()` collection wrapping would have NO place to live.

### Grammar delta
None. This is a pure codegen/emit change; no lexer, parser, or .koi grammar changes. The eligibility predicate is derived entirely from existing AST (ValueObjectDecl.Invariants, MemberAnalysis.IsDerived, IsQuantity) plus the existing OperatorNeedsAnalyzer (_scalarNeeds/_additiveNeeds).

### Migration impact
If adopted: 3 demo VOs change emitted shape (PaymentReceipt, GatewayResult, SalePeriod). Affected files/tests: (a) the intent of GeneratedCodeTests.Value_object_is_a_class_deriving_ValueObject_not_a_record (GeneratedCodeTests.cs:75) — passes today only because it inspects Money (which stays a class); it should be rewritten to positively assert record-ness on an eligible VO or it becomes a misleading test. (b) Any consumer code/snapshots referencing the 3 VOs' equality via the ValueObject base or via GetEqualityComponents. (c) Snapshot tests: the eligible VOs are NOT in the R-fixture snapshots, so EmitterSnapshotTests are unaffected; but the demo (demo/Shop.Domain) regenerated output would change. (d) Docs: website/.../reference/value-objects.md and tutorials/values-and-invariants.md state VOs derive ValueObject — would need a caveat. (e) Behavioral risk: any VO with a list/set/map field would lose by-element equality (none in the current eligible demo set, but the rule must exclude collection-typed VOs to stay correct).

### Rationale
It is NOT a net improvement, for three concrete reasons rooted in the code I read:

1) Structural-equality SEMANTICS regress for any collection-typed field. The whole reason the ValueObject base exists is WriteEqualityComponents wrapping list fields in Ordered() and set/map fields in Unordered() (CSharpEmitter.ValueObjects.cs:160-163) backed by SequenceComponent (Runtime.cs:176). A C# record compares reference-typed members (List<T>, HashSet<T>, Dictionary) by REFERENCE, so two records with equal-by-content collections would compare unequal. SalePeriod wraps Range<Instant> (which has its own value Equals, so it survives), but the general eligible class includes collection-bearing VOs, and a record silently breaks their by-value contract. This is a correctness regression, not cosmetic.

2) It splits one consistent shape into two, hurting taste rather than helping it. Today every VO is `: ValueObject` with GetEqualityComponents — uniform, greppable, polymorphic. The proposal makes ~25% of VOs structurally different (record, no base, `with`, deconstruction) based on a non-obvious predicate (no invariant AND no derived field AND not a scalar/additive target). A reader can no longer tell a VO's shape from `value X { ... }`; they must mentally re-run OperatorNeedsAnalyzer. A VO can also FLIP shape across edits: add one invariant or a derived field and it silently changes from record to class, breaking any consumer using `with` or deconstruction. That instability is worse than the modest boilerplate it removes.

3) Loss of polymorphic base equality. The emitted base defines `operator ==(ValueObject?, ValueObject?)` (Runtime.cs:165). Records don't derive it, so VO-vs-VO comparison through a `ValueObject`-typed reference (or generic constraint) would no longer be value-based for the record subset. The test's intent comment ("not a record, not reference equality", GeneratedCodeTests.cs:62) shows this uniformity was a deliberate design decision.

Payoff is small: only 3 demo VOs, each saving ~5 lines, and the auto-record ToString is already matched by WriteValueObjectToString. The `with`/deconstruction ergonomics are nice but VOs are immutable-by-construction; `with` would bypass invariant guards on any VO that later gains one, which is a footgun in a DDD codebase whose selling point is "can never exist in an invalid state."

If still desired, the only safe variant is: emit records ONLY for VOs that additionally have NO collection-typed fields, AND flip the test from inspecting Money to a positive assertion (`PaymentReceipt` IS a record / IS NOT ValueObject) so intent is explicit — but that increases the divergence problem in (2). Recommend DEFER and keep the uniform ValueObject base.
